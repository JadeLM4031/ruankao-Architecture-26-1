# 案例必杀关键词表

> 案例分析答题核心：**每个问题都要写出具体技术名词，不能只写笼统方向**
> 背熟这张表，案例多拿5-8分

## 一、安全性相关

**问"如何保证安全"时，从这些词中挑3-4个写：**

| 场景 | 必杀关键词 |
|------|-----------|
| 数据传输安全 | **IPSec**（IP层加密）、**SSL/TLS**（传输层加密）、**HTTPS**（HTTP+SSL） |
| 数据存储安全 | **AES-256加密**、**加盐哈希**（SHA-256/bcrypt）存密码、数据库字段加密 |
| 身份认证 | **数字证书**（X.509）、**双因素认证MFA**、**OAuth2.0**、**SAML** |
| 访问控制 | **RBAC**（基于角色的访问控制）、**最小权限原则**、**ABAC**（基于属性的访问控制） |
| 防攻击 | **WAF**（Web应用防火墙）、**参数化查询**（防SQL注入）、**HTML转义**（防XSS）、**Token验证**（防CSRF） |
| 网络安全 | **防火墙**（包过滤/状态检测/应用网关）、**DMZ**（隔离区）、**VPN**、**零信任ZTA** |
| 审计追溯 | **操作日志**、**审计日志**、**安全审计**、**入侵检测IDS** |

## 二、可靠性/可用性相关

**问"如何提高可靠性/可用性"时，从这些词中挑：**

| 场景 | 必杀关键词 |
|------|-----------|
| 硬件冗余 | **双机热备**、**N+1冗余**、**集群部署**、**主从切换** |
| 故障转移 | **自动故障转移（Failover）**、**心跳检测**、**VIP漂移**、**Keepalived** |
| 数据可靠性 | **RAID**（磁盘冗余）、**数据备份**（全量+增量）、**异地灾备**、**主从复制** |
| 服务降级 | **熔断**（Hystrix）、**限流**（令牌桶/漏桶）、**降级策略**、**超时重试** |
| 可用性计算 | **MTBF**（平均故障间隔）、**MTTR**（平均修复时间）、可用性=MTTF/MTBF |
| 高可用架构 | **Nginx负载均衡**、**双活数据中心**、**故障自动切换** |

## 五、数据库相关

## 七、质量属性与架构评估

**问"质量属性/ATAM"时，从这些词中挑：**

| 场景 | 必杀关键词 |
|------|-----------|
| 敏感点 | 某个架构决策对某质量属性的**影响程度**（"XX决策影响性能"） |
| 权衡点 | 影响**多个**质量属性的决策（"增加加密影响安全但降低性能"） |
| 风险 | 可能导致不良后果的架构决策（"存在XX风险"） |
| 效用树 | 根节点→质量属性→属性精化→场景（**四级结构**） |
| 场景六要素 | **刺激源、刺激、环境、制品、响应、响应度量** |

## 八、开发方法与需求

**问"开发模型/需求/测试"时，从这些词中挑：**

| 场景 | 必杀关键词 |
|------|-----------|
| 开发模型选择 | 螺旋=**风险驱动**、RUP=**用例驱动+以架构为中心**、敏捷=**拥抱变化** |
| 测试方法 | **白盒**（逻辑覆盖）、**黑盒**（等价类/边界值）、**回归测试**（修改后重测） |
| 维护类型 | 适应性=**被环境逼着改**、完善性=**用户主动要更多**、纠错性=**修bug** |
| 需求工程 | **功能需求**（做什么）+ **非功能需求**（做得怎样）、**SRS**（需求规格说明书） |

## 答题三步法（每次都用）

```
第一步：直接给结论
"推荐使用XXX技术/架构/方案"

第二步：写具体技术名称和原理
"XXX的核心原理是..."
"具体包括：A、B、C三个组件"

第三步：逐条对应背景条件（最值钱的一步！）
"针对背景①...：XXX可以解决..."
"针对背景②...：XXX提供了..."
"针对背景③...：XXX满足..."
```

**记住**：第三步值2-3分，你之前几乎每次都缺这一步。
