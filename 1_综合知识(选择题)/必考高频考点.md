## ⭐ 2026 新考纲新增考点（第2版教材变化）

> **重要：** 以下为第2版教材新增内容，在2026年考试中必占一定比例，务必重视！

### 新增考点一：ArchiMate 企业架构建模语言

| 维度 | 说明 |
|------|------|
| **是什么** | The Open Group 定义的企业架构建模语言，用于描述、分析和可视化架构 |
| **核心层次** | 业务层(Business) → 应用层(Application) → 技术层(Technology) |
| **与UML区别** | UML 偏向软件设计（类/序列图）；ArchiMate 偏向企业级架构（业务/应用/技术三层） |
| **考试出题点** | ArchiMate 图的识别、与 UML 的对比、三层架构描述 |

### 新增考点二：零信任架构 (Zero Trust Architecture, ZTA)

| 维度 | 说明 |
|------|------|
| **核心原则** | "从不信任，始终验证"（Never Trust, Always Verify） |
| **关键特性** | 以身份为中心、微分段、最小权限、持续验证 |
| **与传统对比** | 传统：边界防御（防火墙=城堡护城河）；ZTA：无隐式信任，每次访问都需认证授权 |
| **考试出题点** | ZTA原则辨析、与纵深防御的区别、适用场景选择 |

### 新增考点三：Lambda 架构与 Kappa 架构（大数据）

| 维比 | Lambda 架构 | Kappa 架构 |
|------|------------|------------|
| **核心思路** | 批处理层 + 速度层 + 服务层（三层） | 统一流处理（只有一层） |
| **优点** | 容错性好、结果准确 | 架构简单、维护成本低 |
| **缺点** | 架构复杂、两套代码维护 | 流处理引擎要求高 |
| **适用场景** | 对准确性要求极高的场景 | 对实时性要求高的场景 |
| **技术栈** | Hadoop(Batch) + Storm/Spark(Streaming) | Kafka + Flink/Spark Streaming |

### 新增考点四：云原生架构

| 核心技术 | 说明 |
|---------|------|
| **容器化** | Docker 等，应用打包隔离运行 |
| **微服务** | 单体拆分为独立部署的小服务 |
| **DevOps** | 开发运维一体化，CI/CD |
| **服务网格(Service Mesh)** | Istio等，微服务间通信管理 |
| **不可变基础设施** | 容器镜像不可变，通过替换而非修改更新 |
| **声明式API** | 描述期望状态而非操作步骤 |

### 新增考点五：AI赋能系统设计

| 要点 | 说明 |
|------|------|
| **大模型架构** | Transformer、注意力机制基础概念 |
| **边缘计算** | AI推理下沉到边缘设备 |
| **MLOps** | 机器学习的DevOps（模型训练-部署-监控全流程） |
| **考试出题点** | 概念辨析为主，不会考深度技术细节 |

## 二、软件架构风格 — 核心考点对比分析

### 2.1 五大架构风格总览

| 架构风格 | 核心思想 | 典型子风格 | 适用场景 | 关键词联想 |
|---------|---------|-----------|---------|-----------|
| **数据流风格** | 数据在构件间流动，逐步变换 | 批处理序列、管道-过滤器 | 数据处理、编译器、信号处理 | 数据驱动、顺序处理 |
| **调用/返回风格** | 构件通过调用-返回机制交互 | 主程序/子程序、面向对象、层次结构 | 通用系统、分层架构 | 函数调用、继承、分层 |
| **独立构件风格** | 构件间无直接调用，通过事件通信 | 进程通信、事件驱动（隐式调用） | GUI、物联网、消息系统 | 解耦、发布/订阅、异步 |
| **虚拟机风格** | 通过虚拟环境解释执行 | 解释器、基于规则的系统 | 专家系统、DSL、AI推理 | 规则引擎、解释执行 |
| **仓库风格** | 以数据中心，构件围绕数据操作 | 数据库系统、黑板系统 | CAD、AI推理、信号处理 | 共享数据、黑板协同 |

### 2.2 逐项深度对比

#### (1) 数据流风格 (Data Flow)

**管道-过滤器（Pipe-Filter）：**
- 每个构件（过滤器）有输入端和输出端，独立处理数据
- 构件之间通过管道传递数据流
- **优点：** 高内聚低耦合、支持复用、支持并行处理
- **缺点：** 不适合交互式应用、数据转换开销大

**批处理序列：**
- 整批数据作为整体在步骤间传递
- 每一步必须完全结束才进入下一步

> **考点陷阱：**
> - "编译器的词法分析→语法分析→语义分析→代码生成"是管道-过滤器的经典例子，**不是**层次结构
> - 管道-过滤器的构件间**不共享状态**，这是区别于仓库风格的关键
> - 注意区分：管道-过滤器可以**并行**执行，批处理是**严格顺序**

---

#### (2) 调用/返回风格 (Call/Return)

**三种子风格对比：**

| 子风格 | 特点 | 考点 |
|-------|------|------|
| **主程序/子程序** | 自顶向下分解，单线程控制 | 最传统的风格，功能分解 |
| **面向对象** | 封装+继承+多态 | **陷阱：** 对象间通过方法调用耦合，修改某对象可能影响调用者 |
| **层次结构** | 每层为上层服务，使用下层服务 | **陷阱：** 层次结构≠管道-过滤器！层次是**调用关系**，管道是**数据流动** |

> **考点陷阱：**
> - "三层架构（表示层-业务层-数据层）"属于**层次结构**，不是数据流风格
> - 面向对象风格的缺点：**对象标识可能变化**、**副作用**（调用某方法改变了对象状态）
> - OSI七层模型、TCP/IP协议栈是层次结构的经典例子

---

#### (3) 独立构件风格 (Independent Components)

**进程通信：**
- 构件间通过消息传递交互（同步/异步）

**事件驱动（隐式调用）：**
- 构件声明感兴趣的事件，事件发生时系统自动调用
- 构件之间**不知道对方的存在**（松耦合）

> **考点陷阱（高频！）：**
> - 事件驱动风格中，构件的调用是**隐式**的，不是显式调用！
> - 典型应用：**观察者模式、发布/订阅系统、IDE（Eclipse）、GUI事件处理**
> - **易错辨析：** "微服务架构通过消息队列通信"更接近独立构件风格，而非调用/返回
> - 事件驱动**不保证执行顺序**——这是和管道-过滤器的关键区别

---

#### (4) 虚拟机风格 (Virtual Machine)

**解释器：**
- 通过解释引擎逐条执行程序
- 核心组件：解释引擎、被解释的程序、程序状态

**基于规则的系统：**
- 使用 if-then 规则进行推理
- 典型应用：**专家系统、业务规则引擎、AI推理系统**

> **考点陷阱：**
> - "专家系统"属于**虚拟机风格**（基于规则的系统），**不是**独立构件！
> - 解释器风格的**灵活性高但性能低**，这是常见出题点
> - Prolog语言、SQL解释执行都可以归类为虚拟机风格
> - 注意区分：解释器是**逐条执行**，编译器是**整体翻译**（编译器前端=数据流风格）

---

#### (5) 仓库风格 (Repository)

**数据库系统：**
- 中央数据存储，客户端通过查询/更新操作数据

**黑板系统：**
- 三大核心构件：**黑板（共享数据）、知识源（独立处理模块）、控制器（调度策略）**
- 适用于：**没有确定性解法的复杂问题**（如语音识别、图像处理、AI推理）

> **考点陷阱（高频！）：**
> - **黑板系统的三大构件必须记住**——案例题可能直接问
> - "多个专家协同解决问题"是黑板系统的典型场景
> - 黑板系统 vs 事件驱动：黑板是**共享数据驱动**，事件驱动是**事件触发驱动**
> - **陷阱题：** "编译器的符号表管理"涉及仓库风格（符号表=共享数据仓库）

---

### 2.3 架构风格识别 — 秒杀口诀

| 关键词 | → 对应风格 |
|-------|-----------|
| 管道、过滤、数据流、逐步变换 | 数据流 |
| 分层、层次、调用、继承 | 调用/返回 |
| 事件、发布/订阅、隐式、解耦 | 独立构件 |
| 解释、规则引擎、专家系统 | 虚拟机 |
| 共享数据、黑板、中心数据库 | 仓库 |

---

### 2.4 架构风格选择题 — 常见出题套路

**套路1：给场景选风格**
> "某系统需要将数据经过多个步骤逐步处理，每个步骤独立，支持增减步骤" → **管道-过滤器**

**套路2：给风格选缺点**
> "面向对象架构的主要缺点" → **对象间耦合、标识可能变化、副作用**

**套路3：概念辨析**
> "以下哪个不是管道-过滤器的特点" → 选项中混入层次结构或事件驱动的特点

**套路4：综合应用**
> "某系统包含多个知识源，通过共享数据区协同工作" → **黑板系统（仓库风格）**

## 三、其他高频考点速记

### 3.1 系统开发方法

| 开发模型 | 核心特点 | 适用场景 |
|---------|---------|---------|
| 瀑布模型 | 线性顺序、文档驱动 | 需求明确、变更少 |
| 原型模型 | 快速构建原型确认需求 | 需求不明确 |
| 螺旋模型 | 风险驱动、迭代 | 大型复杂项目、高风险 |
| 增量模型 | 分批交付、逐步完善 | 需要快速交付部分功能 |
| 喷泉模型 | 面向对象、迭代无间隙 | OO开发项目 |
| 敏捷方法 | 拥抱变化、人本主义 | 需求变化频繁 |

### 3.2 数据库 - 范式必记

| 范式 | 要求 | 消除的依赖 |
|-----|------|-----------|
| 1NF | 属性不可再分 | 消除非原子值 |
| 2NF | 消除部分函数依赖 | 非主属性不完全依赖于码 |
| 3NF | 消除传递函数依赖 | 非主属性不传递依赖于码 |
| BCNF | 每个决定因子都是候选码 | 主属性对码的部分/传递依赖 |

### 3.3 设计模式 - 高频TOP 6

| 模式 | 类型 | 一句话描述 | 考频 |
|-----|------|-----------|:---:|
| 单例(Singleton) | 创建型 | 保证一个类只有一个实例 | ★★★★★ |
| 工厂方法(Factory Method) | 创建型 | 子类决定实例化哪个类 | ★★★★ |
| 观察者(Observer) | 行为型 | 一对多依赖，状态变化通知 | ★★★★★ |
| 策略(Strategy) | 行为型 | 算法族封装，可互换 | ★★★★ |
| 适配器(Adapter) | 结构型 | 接口转换，兼容已有类 | ★★★★ |
| 代理(Proxy) | 结构型 | 控制对象访问 | ★★★☆ |
